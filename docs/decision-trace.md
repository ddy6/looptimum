# Decision Trace and Acquisition Log

This document explains what Looptimum records for each suggestion, how to read
the decision trace, and what the fields mean.

## Where the Trace Lives

Decision traces are written as JSON Lines:

- `state/acquisition_log.jsonl`

Each line corresponds to one `suggest` event.

Related runtime log:

- `state/event_log.jsonl` tracks lifecycle/ops events (`cancel`, `retire`,
  `heartbeat`, lock events, report generation) and is intentionally separate
  from acquisition decision records.

Phase 6 reference artifacts:

- `docs/examples/decision_trace/golden_acquisition_log.jsonl`
- `docs/examples/decision_trace/golden_acquisition_log.md`
- `docs/examples/decision_trace/cli_transcript.md`

## Record Shape

Each log line has the same top-level structure:

```json
{
  "trial_id": 7,
  "decision": {...},
  "timestamp": 1772392830.728097
}
```

Top-level fields:

- `trial_id`: integer trial identifier generated by `suggest`
- `decision`: strategy and scoring metadata
- `timestamp`: Unix epoch seconds when decision was logged

## Decision Object Fields

| Field | Meaning | Present When |
|---|---|---|
| `strategy` | Selection path used for suggestion | Always |
| `surrogate_backend` | Model backend label (`rbf_proxy`, `gp`, `botorch_gp`, or `null`) | Template-dependent |
| `acquisition_type` | Heuristic type (`ei`, `ei_proxy`, or `ucb`) | Surrogate-acquisition mode |
| `predicted_mean` | Predicted objective at candidate | Surrogate-acquisition mode |
| `predicted_std` | Predicted uncertainty proxy | Surrogate-acquisition mode |
| `acquisition_score` | Ranking score used to pick candidate | Surrogate-acquisition mode |
| `fallback_reason` | Error string when GP path falls back to proxy | Optional in `bo_client_full` |

## Strategy Semantics

### `initial_random`

- Used during warmup before enough observations are available.
- Candidate comes from bounded random sampling.
- Usually logged without predicted mean/std/acquisition score fields.

### `surrogate_acquisition`

- Used after warmup.
- Candidate pool is sampled, scored, and ranked.
- Highest acquisition score candidate is emitted.

## Backend Notes by Template

### `templates/bo_client_demo`

- Uses proxy scoring only.
- `decision` includes strategy and acquisition fields in acquisition mode.
- `surrogate_backend` is not consistently included for all records.

### `templates/bo_client`

- Supports `rbf_proxy` and optional `gp` backend via config.
- Decision records include `surrogate_backend` in both warmup and acquisition
  modes.

### `templates/bo_client_full`

- Supports proxy and optional BoTorch GP path (feature flag).
- If GP path is unavailable and fallback is enabled, the decision records
  `fallback_reason`.

## Acquisition Heuristic Summary

Current public templates use heuristic scoring over candidate pools:

- `ucb`: exploration-adjusted optimistic score
- `ei` / `ei_proxy`: expected-improvement style score

Proxy backend details:

- Surrogate mean/std are estimated using an RBF-distance weighted proxy over
  historical observations.

GP backend details:

- `bo_client`: optional GP path in `surrogate_gp.py`.
- `bo_client_full`: optional BoTorch `SingleTaskGP` path with proxy fallback.

## How to Audit a Run

Recommended audit flow:

1. Read `bo_state.json` for canonical pending/observed/best state.
2. Read `acquisition_log.jsonl` chronologically for suggestion decisions.
3. Match `trial_id` across suggestion, ingest payloads, and observations.
4. Confirm strategy transitions (warmup to surrogate-acquisition) are expected.
5. Use `fallback_reason` to identify dependency/environment backend issues.

## Reproducibility Boundaries

Useful anchors:

- Trial ids and suggestion order are deterministic for a fixed state/config path.
- Decision logs provide full sequence order for post-mortem analysis.

Known variability:

- Timestamp values are wall-clock dependent.
- GP behavior may vary with dependency versions and numeric backend details.
- External evaluator noise can shift downstream decision trajectories.

## What This Trace Is and Is Not

The decision trace is:

- an auditable record of suggestion-time metadata
- useful for debugging backend selection and acquisition behavior

The decision trace is not:

- a formal proof of global optimality
- a substitute for problem-level validation and domain review
